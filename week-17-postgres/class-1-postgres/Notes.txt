Types of Databases:
1. NoSQL databases: Store data in a schemaless fashion. 
We define the schema, but MongoDB is inherently schemaless. Mongoose enforces the schema at the Node.js level.
2. Graph Databases: Neo4j (Useful for Social Networks).
3. Vector Databases: Useful in Machine Learning, text-> embedding model -> objects as vector -> vector database. Examples: Pinecone.
4. SQL Databases: Store data in the form of rows. Most full-stack applications use this. Examples: MySQL, Postgres (preferred).

-------------------------------------------------------------------------------------------
Schema can be changed in SQL database as app changes.
Migrations happens when schema is changed in SQL database.

Steps to use a SQL database:
1. Run the database.
2. Create a table and define its schema.
3. Using a library, connect to the DB and put data.
4. Run queries on the database to interact with the data (insert/update/delete).

Create Table Command:
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  username VARCHAR(50) UNIQUE NOT NULL,
  email VARCHAR(255) UNIQUE NOT NULL,
  password VARCHAR(255) NOT NULL,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

People usually change thier username/email, so it's not a good idea to make them Primary Keys.
Auto-incremental count keeps running even for wrong/invalid queries.

Another way:
CREATE TABLE Users (
  id INT,
  username TEXT NOT NULL UNIQUE,
  CONSTRAINT pk_id PRIMARY KEY (id)
);

Now we have to create an application that connects to the database. 
We can create a todo application, an express application, HTTP server, or websocket server. 
Let's create simple functions which connect to the database.

import { Client } from "pg";
const client = new Client("connection_string");
await client.connect();
console.log("Connected to database");
const response = await client.query("SELECT * FROM Users");
console.log(response.rows);
await client.end();

Code:

app.post("/signup", async (req, res) => {
  const username = req.body.username;

  // const query = `INSERT INTO users (username, password) VALUES ('${username}', '${password}');`;
  // await pgClient.query(query); 
});

Here we are blindly sending the username and password to the database, but a malicious user can send a string that closes the current query and executes another query on the database, potentially deleting something.
Example malicious input: 
`'); DELETE FROM users; INSERT INTO users (username, password) VALUES ('user', 'pass`

To avoid this, we send parameters via variables:
await pgClient.query(
  "INSERT INTO users (username, password) VALUES ($1, $2)", 
  [username, password]
);

Q&A 17.1
1. What is Connection Pooling in Neon DB?
Connection pooling is a database optimization technique where instead of opening and closing a physical database connection for every single user request, we maintain a cache of active, long-lived connections and reuse them.

Creating a fresh database connection from scratch is a highly expensive operation because it involves a network handshake, authentication, and memory allocation on the database server. In high-traffic or serverless environments (like AWS Lambda or Neon DB), thousands of rapid requests hitting the database directly would instantly exhaust its memory limits and crash the system.

With pooling, the application simply borrows an already-open, pre-warmed connection from the pool, executes the query instantly, and returns it back to the pool upon completion. A good analogy is a Taxi Stand—instead of manufacturing a new car every time someone wants to travel, passengers just catch an available cab, complete their journey, and return it to the stand for the next passenger.

2. How do we scale SQL databases?
- Vertically: Add more resources to the machine.
- Horizontally: Master-Slave architecture. 
You have more reads than writes in SQL databases. (Master node for write, Slave nodes for read).
- Sharding is scaling by dividing your application data, but joins are difficult across shards.

Good architecture for a chatting application using websockets:
- You should scale it.
- Image/file uploads.
- Threads/nested threads (someone can reply to you).
- Tagging (when tag happens, show existing users).
- Group chats.
- Dynamic chats (create own group).
- If a group has 50 people, how does a message reach those 50 people? (Kafka).
